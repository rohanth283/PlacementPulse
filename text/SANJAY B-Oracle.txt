You are asked to design a data model to represent a large, dynamic 
class hierarchy with the following properties. Code implementation 
of the entire class structure and functions was asked. 
 
Root Class: 
  
There is a single base class called Node. 
Alphabetical Branching: 
  
Each class can have up to 26 direct subclasses, one for each 
letter A through Z. 
  
Each of those subclasses can themselves have A–Z subclasses, 
and so on to arbitrary  depth 
Inheritance Rules: 
    
Multiple inheritance is allowed, but the design must prevent the 
“diamond problem” (no class should inherit from the same ancestor 
along two different paths).In other words, a class may inherit from 
more than one parent, but the resulting graph must remain a directed 
acyclic graph (DAG) with no converging cycles. 
Dynamic Updates: 
  
Classes (nodes) may be added, removed, or modified at any 
time. 
  
The structure must handle these operations efficiently without 
downtime. 
Class Metadata: 
   
Each class can define its own set of methods, properties, and 
other metadata. 
 
Required Operations 
1. Fast Retrieval 
Quickly locate a specific class by its full hierarchical path (e.g., Node → 
C → F → …). 
2. Efficient Search 
Support lookups by name or prefix (e.g., “find all classes whose path 
starts with AB”). 
Handle queries such as “find all subclasses that override method X.” 
3. Traversal and Batch Processing 
Enumerate or update all subclasses within any subtree or inheritance 
branch. 
4. Dynamic Maintenance 
Support real-time insertion, deletion, and re-linking of inheritance 
relationships  
 
My Approach & Code was written in Python: 
My solution for the  system as a Directed Acyclic Graph (DAG), where each 
class is a graph node. A node stores its name, its methods, links to its 
parents and children, and also a cached set of all ancestors for fast 
validation. To make class lookup efficient, we maintain a Trie (prefix tree) 
where each path like "A/B/C" maps directly to the corresponding class node. 
This gives fast lookups by path and also makes prefix-based searches easy. 
To prevent the diamond problem, whenever we add a new inheritance link, 
we check two things: 
1. No cycles are allowed (a class cannot become its own ancestor). 
2. No ancestor overlap between parents—if two parents share a common 
ancestor, adding this link would create a diamond, so we reject it. 
3. The structure supports dynamic updates: 
• 
Adding a class means creating a new node, inserting it into the trie, 
and linking it under a default root.Adding inheritance requires running 
the cycle and overlap checks, then updating ancestor sets.Removing a 
class just means unlinking it from its parents/children and updating 
descendants. 
• 
For retrieval and search, path lookups are done by following the trie, 
prefix-based searches are just trie traversals, and traversals like 
BFS/DFS can be run directly on the graph of nodes. 
Graph (DAG) + Trie + Ancestor checks
